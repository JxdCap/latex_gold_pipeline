# Repository Guidelines

## 项目定位
本仓库用于中文 LaTeX 语料清洗与分桶，不是通用数据工程模板。工作边界只覆盖语料导入、轻预洗、分桶、分流，不扩展到脱壳、金标转换或后续标注生产。

## 数据入口与基础约束
唯一默认入口是 `data/输入表格/input.xlsx`。处理时必须先导入 `input.xlsx`，再转为 `jsonl`，不能跳过这一步直接从其他中间格式起跑。

不确定的样本不要强判，统一打标并留到后续复核。所有批处理都先跑小样本，再跑全量，避免一次性污染整批数据。

## 固定工作顺序
所有实现、脚本和人工操作都按以下顺序执行，不能调换：

1. 导入
2. 轻预洗
3. 分桶
4. 分流

`轻预洗` 只做低风险整理，例如去掉明显空行、统一基础字段、补齐可直接推断的结构信息。不要在这一阶段引入激进规则。

## 每阶段输出要求
四个阶段默认都同时输出 `jsonl` 和 `xlsx` 两份结果，便于程序消费和人工抽检。命名应体现阶段与批次，例如：

- `data/import/sample.jsonl`
- `data/import/sample.xlsx`
- `data/bucket/full.jsonl`
- `data/bucket/full.xlsx`

如果只产出单一格式，视为流程未完成。

## 目录与实现建议
- `src/`: 导入、清洗、分桶、分流脚本。
- `tests/`: 针对各阶段的小样本验证。
- `configs/`: 桶规则、字段映射、打标枚举。
- `data/`: 输入、阶段性产物与抽检样本。

新增模块时优先按阶段拆分，例如 `src/import_excel.py`、`src/preclean.py`、`src/bucket.py`、`src/route.py`，避免把整条链路堆进一个脚本。

## 开发与校验要求
- 先用小样本跑通 `input.xlsx -> jsonl -> 轻预洗 -> 分桶 -> 分流` 全链路，再处理全量。
- 每次改规则后，至少复查一个 `jsonl` 输出和一个 `xlsx` 输出。
- 测试命名建议使用 `test_<stage>.py`，例如 `test_bucket.py`。
- 提交信息应直接说明阶段和改动，例如 `调整分桶规则`、`补充预洗打标逻辑`。

## 禁止事项
- 不要把项目范围扩展到脱壳。
- 不要在本仓库内加入金标转换流程。
- 不要覆盖原始 `input.xlsx`。
